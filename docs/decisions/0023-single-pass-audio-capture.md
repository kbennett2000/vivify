# ADR-0023: Single-pass synthesis via null-sink monitor capture; close the teardown gap with a fast exit
Status: Accepted · Date: 2026-06-21

## Context
Cycle 11 (`docs/cycles/cycle-11-latency-singlepass.md`, branch `cycle-11-latency-singlepass`) continues the latency work, scoped to the **server + bridge only** (`services/voice-server`); `@vivify/core`, `@vivify/acs`, and the IR are untouched.

PR #12's per-stage instrumentation (ADR-0022) measured a steady-state Speak at `total=7960ms`: init ~18ms + Pass A ~2900ms + Pass B ~2850ms + a ~2170ms unexplained gap (server total minus bridge self-time). This isolated two avoidable costs on top of the inherent floor:

- **The redundant second pass.** The bridge was two-pass (ADR-0019): Pass A (`CLSID_MMAudioDest`, real-time) for the dense, accurate viseme events, and Pass B (`CLSID_AudioDestFile`) purely to produce the WAV — because `MMAudioDest` cannot tee its rendered PCM (the ADR-0019 finding). Pass B is a full second synthesis (~2.8s) whose only output is audio Pass A already rendered.
- **The ~2170ms gap.** Wall time inside the child process but outside `main()`'s self-measured window: the Wine process LOAD (spawn → `main`) and TEARDOWN (after the timing print → process exit: `CoUninitialize` + DLL unload + device close/drain).

A side diagnostic — "Pass A 1100ms on warmup vs 2900ms steady, looks backwards" — is **resolved, no action**: Pass A is real-time playback, so its duration tracks utterance length; the warmup phrase ("warm") is far shorter than the steady test sentence.

## Decision

**1. Single pass — capture the null-sink monitor instead of synthesizing a second time (kill Pass B).**
The bridge is now single-pass: Pass A (`MMAudioDest`, real-time) only. While Pass A plays in real time, the server records the PulseAudio null sink's `.monitor` source with `parec`, so one real-time pass yields BOTH the dense viseme events AND the audio. The bridge no longer writes a WAV (`--wav` is accepted but ignored). The server orchestrates the capture via a new injectable `captureCommand` (default `parec --device=dummy.monitor --format=s16le --rate=44100 --channels=1`), wraps the captured RAW PCM into a RIFF/WAVE in Node (`src/wav.ts` `wrapPcmToWav`), and trims leading silence (`trimLeadingSilence`) so the WAV's t=0 aligns with the timeline's first viseme. The null sink format is pinned (`pulse-null.pa`: s16le / 44100 / mono) so the monitor capture is deterministic — no resample guesswork. The `{ audioWavBase64, mouthTimeline }` contract is unchanged. Expected saving ≈ Pass B (~2.8s).
- **Why RAW PCM, not a `parec`-written WAV:** there is no WAV header to finalize, so even an abrupt `SIGTERM` stop yields valid samples — Node owns the header.
- **This supersedes ADR-0019's two-pass approach for the WAV source only.** The dense-events requirement (real-time `MMAudioDest`) is unchanged; only the audio source changes — monitor capture instead of a second `AudioDestFile` pass.
- **HONEST FAILURE (a project non-negotiable — no faking):** if the capture yields no audio (empty PCM) or only silence (trims to empty), the server returns a loud 500, NOT a silent or faked WAV. The real capture path is unverifiable in the sandbox (no Wine/PulseAudio in CI); it is operator-verified, and the directive was explicit — if the capture/format/alignment fights, STOP and report rather than fake the audio.

**2. Close the ~2170ms gap: locate it, then fast-exit the bridge.**
To locate the split, the bridge prints a `[boot]` line as `main()`'s first statement and the server timestamps the child's first stderr byte, so `wineLoadMs ≈ firstByte − spawn`. Combined with `bridgeMs` (spawn → close) and the bridge's own `totalMs`, the gap decomposes into load / self / teardown, all surfaced in `[tts-timing]` (`src/timing.ts`: added `wineLoadMs` / `captureMs`, computed teardown). To CLOSE the cheap part, the bridge does `fflush(NULL); _Exit(rc)` after writing the timeline, skipping the graceful COM/DLL/device teardown that sits after the timing print — safe for a one-shot process, since the OS/Wine reclaims everything. Single-pass also removes one `MMAudioDest` open/close cycle.
- **RESIDUAL / future work:** if a large `wineLoadMs` (the spawn → `main` prologue) remains, the only further fix is a persistent-engine bridge daemon (keep the Wine process alive, synthesize on demand) — a large rewrite, deliberately OUT OF SCOPE this cycle and reported with its measured number. This is the remaining structural cost.

### Amendment 2026-06-21 (fixing Cycle 11 / PR #13): gate synthesis on the capture being live.
On-screen testing of the single-pass path revealed the audio was CLIPPED at the start — the first ~2–3s of speech was missing and WAV sizes varied per request: a classic start race. Root cause: the server spawned `parec` and then immediately spawned the bridge, but **spawning `parec` ≠ `parec` streaming** — `parec` has connect/startup latency, so the engine reached real-time playback before the null-sink monitor was actually being recorded, losing the opening audio.

**3. Wait for the capture's first PCM sample before triggering synthesis.**
The server now blocks on proof that the monitor is open and flowing — the capture's first `stdout` chunk — BEFORE it spawns the bridge. Synthesis is gated on this readiness.
- **Capture-readiness gate:** resolves on the first `stdout` chunk; rejects if no sample arrives within `captureReadyTimeoutMs` (default 5000ms, env `VIVIFY_CAPTURE_READY_MS`) or the capture process dies first. A reject is a loud 500 — NEVER a clipped, silent, or faked WAV (the honest-failure boundary again).
- **Timing-independent by construction:** gating on first-sample makes the result independent of capture-start latency. The leading silence between capture-live and the engine's `AudioStart` is then removed by `trimLeadingSilence` — which is also why WAV sizes now stabilize across requests.
- **Why the gate resolves fast instead of deadlocking:** the null sink never suspends (no `module-suspend-on-idle` loaded), so its `.monitor` streams continuous silence the instant `parec` connects. `parec --latency-msec=30` makes that first sample flush promptly, so the gate resolves in ~tens of ms and adds little to the ~5300ms total.
- **`leadInMs` guard on `trimLeadingSilence`** (default 40ms): keeps a small margin before the detected onset so a soft leading consonant isn't shaved. It aligns the WAV's t≈0 to the timeline's first viseme safely, rather than trimming flush to the first loud sample.

### Amendment 2026-06-21 (Cycle 11 follow-up / PR #13): skip Wine's teardown by kill-after-done, and warm the whole capture pipeline.
Post-gate measurement: `total=6011ms (captureReady=747 bridgeWall=5055 wineLoad=163 teardown=2000 capture=6004 encode=0)`, bridge `[init=8 passA=2873 write=0 self=2892]`. Two avoidable costs remained: a ~2000ms `teardown` (`bridgeWall − wineLoad − self`) that `_Exit` did NOT eliminate, and a still-clipped FIRST Speak after container start (also a high `captureReady=747`). The fixes are **server-side**, plus a one-line bridge hardening: the bridge now emits the `[timing]` line **only when `rc == 0`**, making it the kill-trigger's definitive *success* marker (it was printed unconditionally, so a write-failure path could otherwise have been killed-as-success). On branch `cycle-11-latency-singlepass`.

**4. SIGKILL the bridge the instant it's logically done, instead of waiting on Wine's process teardown.**
The 2000ms `teardown` is wall-time between the bridge printing its final `[timing]` stderr line and the process actually closing: Wine's kernel-side process teardown (winepulse audio-device close/drain + DLL detach) blocks ~2s, and it sits OUTSIDE the C runtime — so the bridge's `fflush; _Exit(rc)` (Decision 2) cannot skip it. But by the time `[timing]` prints, the bridge's useful work is already complete: the timeline file is written AND `fclose`d (before `[timing]`), and all audio has already played to the null sink. So the server now watches the bridge's stderr and, the instant it sees `[timing]`, sets `killedAfterDone` and calls `child.kill('SIGKILL')`. The `close` handler treats a kill-after-done as SUCCESS; a real FATAL *before* `[timing]` still exits non-zero → 500, unchanged. The kernel kills the process so `close` fires immediately and the ~2s Wine teardown happens async, off the request's critical path. Reading the timeline after the kill is safe — it was already closed. Expected: `teardown → ~0`, `total → ~4000ms`.
- **RESIDUAL RISK (record + STOP-and-report boundary):** the next request opens a fresh `MMAudioDest` stream. If force-killing leaves the null sink / winepulse in a state that fails the next `Select`, that is an operator-visible honest failure — the bridge logs the HRESULT and we stop and report, not paper over it.

**5. Warm the WHOLE capture pipeline at startup, not just the engine.**
The first real Speak still clipped because the entrypoint warmup ran a bridge-ONLY synth: it primed the engine/DLLs but never exercised `parec` / the null-sink monitor / winepulse, so the capture path cold-started on the first request (also why `captureReady` was a high 747ms). Fix: a new `warmUp()` in the server runs ONE real `synthesize()` — the exact production path (`parec` connect + monitor + winepulse playback + trim + engine) with a dummy phrase — fired in the background from `start()` after `listen`, best-effort (swallows errors, logs `[warmup] …`). The redundant bridge-only warmup is removed from `entrypoint.sh`; the server warmup supersedes it and also warms the engine.
- **RESIDUAL:** if `captureReady` stays high after warming, a persistent / pre-connected `parec` is the deferred next step.

### Amendment 2026-06-21 (Cycle 11 follow-up, second pass / PR #13): resolve on `[timing]`, do NOT await the bridge's `close`; honest per-step clip metrics; keep-warm monitor reader.
Decision 4 above was **INCOMPLETE and is corrected here.** A warm `--no-cache` rebuild measured `total=5769ms (captureReady=506 bridgeWall=5058 wineLoad=163 teardown=2002 capture=5766 encode=1)`, bridge `[init=8 passA=2873 write=1 self=2893]` — `teardown=2002` was UNCHANGED despite Decision 4's "SIGKILL on `[timing]`", and the first Speak still clipped. These fixes are **server + entrypoint only** (no bridge C++ change), on branch `cycle-11-latency-singlepass`.

**6. CORRECTION to Decision 4 — resolve the bridge-wait promise the instant `[timing]` arrives; do NOT wait for the child's `close` event.**
Decision 4 SIGKILLed the bridge on `[timing]`, but the server still RESOLVED its bridge-wait promise only inside the `close` handler — so it kept blocking ~2s anyway, and `teardown` never moved. Root cause: SIGKILL kills the `wine` launcher, but the underlying Windows process (under the persistent `wineserver`) lingers ~2s in teardown holding the stderr pipe open, so Node's `close` fires ~2s late. (The prior CI test passed FALSELY: killing a Node fake closes its pipe instantly, unlike Wine — so the test never exercised the lingering-pipe case the real bridge exhibits.) So Decision 4's diagnosis was wrong on two counts: the 2s was NOT Wine being unkillable, and it was NOT parec-stop (parec stop measures fast) — it was purely the server awaiting `close`. The fix: the server now RESOLVES the moment a complete `[timing] …\n` line lands in the stderr buffer (by then everything is done — timeline written+`fclose`d, audio played), then SIGKILLs **and** `unref()`s the child in the BACKGROUND, awaiting nothing. The ~2s Wine teardown now happens entirely off the request's critical path. The `close` handler is demoted to covering only the failure / early-exit path (FATAL before `[timing]` → 500). Expected: `teardown → ~0`, `total → ~3500–4000ms`.

**7. Per-step diagnostics to LOCATE the clip honestly, not guess it.**
`[tts-timing]` now also reports `captureStop` (parec SIGTERM→close) and `build` (wrap+trim), and a second `[tts-audio]` line reports `wavMs` (final WAV audio duration) vs `timelineMs` (mouth-timeline span) vs `rawCaptureMs` / `trimmedMs`. A clipped opening is now QUANTIFIED — `wavMs ≪ timelineMs` ⇒ that many ms of audio are missing — rather than inferred from "it sounds cut off."

**8. Keep-warm monitor reader for the first-Speak clip.**
The first real Speak still clipped after `[warmup] done` because each `/tts` spawns a FRESH `parec`, and the null-sink monitor goes cold/idle between the warmup's `parec` and the first real `parec`. Fix: `entrypoint.sh` now starts ONE long-lived `parec -d dummy.monitor >/dev/null &` for the container's lifetime, keeping the monitor source continuously active so every per-request `parec` connects to a hot monitor (no cold-start clip; smaller `captureReady`).
- **RESIDUAL / HONEST CAVEAT:** this targets the parec/monitor-cooling cause. If the clip is instead bridge-side **winepulse PLAYBACK** cold-start, the keep-warm reader will NOT fix it — but `warmUp()` (Decision 5) exercises a full real bridge, and the new `[tts-audio]` clip metric will reveal which side it is. If it persists, a persistent continuous-`parec` with rolling-buffer windowing is the deferred next step.

### Amendment 2026-06-21 (Cycle 11 follow-up / PR #13): persistent capture source + windowing replaces per-request `parec` and the readiness gate; operator-tunable leading-edge trim. SUPERSEDES Decision 3 (the capture-readiness gate) and Decision 8 (the keep-warm reader).
A warm, post-teardown-fix measurement showed `teardown=0` held (good) but exposed two residuals: `total=3869ms (captureReady=608 …)` then `total=5029ms (captureReady=1765 …)` for the SAME phrase — `captureReady` swung 608↔1765ms — and the FIRST Speak still clipped the opening. The `[tts-audio]` line read `wavMs=2983 timelineMs=2212 rawCaptureMs=3267 trimmedMs=284`. These fixes are **server-only** (no `@vivify/core`/browser, no bridge C++ change), on branch `cycle-11-latency-singlepass`.

**9. Persistent capture source + per-request windowing replaces the per-request `parec` AND the readiness gate (SUPERSEDES Decision 3, and SUPERSEDES Decision 8's keep-warm reader).**
`captureReady` was the per-request `parec`'s spawn→first-sample latency; spawning a FRESH `parec` client per `/tts` (process startup + PulseAudio stream setup, worse while the previous request's backgrounded Wine teardown churns) is inherently variable — hence the 608↔1765ms swing. Fix: ONE long-lived `parec` reads the null-sink monitor for the container's lifetime (new `src/capture.ts` `CaptureSource`), and the server WINDOWS its stream per request — `beginWindow()` before the bridge, `endWindow()` after grace returns the captured PCM. No per-request spawn ⇒ no `captureReady` variance (replaced by `windowFirstByteMs`, consistently ~tens of ms), and because the capture is ALWAYS-LIVE the original start-race (Decision 3's reason for existing) stays solved WITHOUT a per-request gate — so the readiness gate is removed. PCM is buffered ONLY during an open window (idle silence discarded → bounded memory). The source respawns if it dies; an empty window → honest 500 (the honest-failure boundary, unchanged). This also SUPERSEDES Decision 8's entrypoint keep-warm reader: the persistent `CaptureSource` IS the continuous reader, so a separate keep-warm `parec` is redundant. `/tts` is now SERIALIZED (one window at a time) — concurrent synth was already broken by the shared null sink, so serialization is strictly safer and additionally removes the warmup-vs-first-request race.

**10. Trim leading-edge fix for the first-Speak clip — the metric proved it was the TRIM, not the capture.**
`wavMs(2983) > timelineMs(2212)` means the audio IS fully captured (the WAV is LONGER than the timeline), and `trimLeadingSilence` shaved `trimmedMs=284` off the FRONT — on a cold/soft opening, into the first consonant. The null-sink monitor is digital silence (zeros) when idle, so the onset threshold can be low without false onsets. Changed `trimLeadingSilence` defaults: threshold 512→150 (detect soft onsets accurately), minRun 4→3, leadInMs 40→80 (safe cushion). Because the WAV is always longer than the timeline, the extra lead-in costs no real speech. Added OPERATOR ENV KNOBS `VIVIFY_TRIM_THRESHOLD` / `VIVIFY_TRIM_LEADIN_MS` so the operator can dial leading-edge alignment WITHOUT another code round — the real audio is unverifiable in CI.

### Amendment 2026-06-21 (Cycle 11 follow-up / PR #13): gate the startup warmup on the reader being LIVE.
The persistent-source amendment claimed the startup `warmUp()` absorbs the cold first-window so the user's first Speak is warm — but the operator's log showed `[warmup] failed … reader not live` printing BEFORE `[capture] … is live`, i.e. the warmup was still racing the reader. Diagnosed via diff: `createVoiceServer` fired the warmup synthesis the instant after `source.start()`, with no wait for the persistent `parec` to actually be streaming, so its window was empty → it threw `null-sink capture window was empty` and no-op'd. (The equivalent of the per-request `await captureReady` that protected earlier commits was dropped with the gate, and nothing replaced it for the warmup.)

**11. Wait for the persistent reader's first sample before the warmup synthesis.** `CaptureSource` gains `whenLive(timeoutMs): Promise<boolean>` — resolves `true` at the moment the reader produces its first PCM sample (where it logs `… is live`), or `false` on timeout (best-effort, never hangs; resets on reader death). `createVoiceServer` now `await source.whenLive(captureReadyTimeoutMs)` INSIDE the serialize mutex before running the warmup synth, so the warmup opens a real, non-empty window and the first real `/tts` (also serialized) queues behind it. Ordering is now instrumented (`[warmup] awaiting capture reader…` → `[warmup] reader live after Nms — priming…` → `[warmup tts-audio] …` → `[warmup] done`, or `[warmup] skipped — reader not live within …ms`) so the operator can SEE the sequence. **Open question this surfaces honestly:** if the null-sink monitor only streams *while audio plays* (not when idle), `whenLive` cannot resolve before the first playback and the log will read `[warmup] skipped` — which would prove a *priming read/write* (keeping the sink running independent of a request) is the real fix. The instrumentation distinguishes these without further guessing.

## Consequences
- **Latency budget after this cycle:** Pass A's real-time synthesis remains the inherent floor (unchanged). Pass B (~2.8s) is eliminated by overlapping audio capture with Pass A. The teardown portion of the gap is closed via `_Exit`. `wineLoadMs` is the named residual, addressable only by a persistent-engine daemon (deferred).
- **New failure mode:** capture problems now fail LOUDLY (a 500) rather than silently degrading into a quiet or empty WAV. Fidelity over graceful degradation, per the constitution.
- **Verification boundary is honest.** `src/wav.ts`'s pure functions (`wrapPcmToWav`, `trimLeadingSilence`) and the server orchestration flow with an injected fake capture ARE CI-tested. The real PulseAudio monitor capture and the single-pass bridge (C++) are NOT runnable in the dev sandbox (no Wine/PulseAudio); they are operator-verified on rebuild, with the real numbers landing in the cycle doc.
- **Teardown is now off the critical path (amendment 2026-06-21).** Kill-after-done moves Wine's ~2s process teardown async; the request returns the instant `[timing]` prints. The first real Speak behaves warm (audible opening) because `warmUp()` exercises the full capture path at startup, and steady-state `captureReady` should drop. New assumed dependency: that force-killing the bridge leaves the null sink / winepulse healthy for the next request — if not, the next `Select` fails loudly (HRESULT logged) rather than degrading. Both are operator-verified on rebuild (no Wine/PulseAudio in CI); CI covers the teardown-skip deterministically via a fake bridge that prints `[timing]` then sleeps ~3s (the server returns in <1.5s, proving it kills rather than waits) and a `warmUp` test that it runs one synth and swallows failures.
- **Teardown is ACTUALLY off the critical path now (amendment 2026-06-21, second pass).** Decision 4 alone did not move `teardown` (measured 2002ms unchanged) because the server still awaited the child's `close`; the fix resolves on the `[timing]` stderr line and backgrounds the SIGKILL + `unref()`, so the ~2s Wine teardown no longer blocks the response (`total → ~3500–4000ms`). The clip is now QUANTIFIED via the new `[tts-audio]` (`wavMs` vs `timelineMs`) and `[tts-timing]` (`captureStop`, `build`) metrics rather than guessed, and a long-lived `parec` keep-warm reader keeps the null-sink monitor hot between requests. Verification boundary: CI proves the resolve-on-`[timing]` fix deterministically via a NEW fake bridge that, after printing `[timing]`, keeps its stderr pipe open ~3s via a DETACHED grandchild (replicating Wine — unlike the old fake whose pipe closed instantly on SIGKILL, which is why the old test passed falsely); the server returns in <1.5s. The keep-warm reader, real teardown, and cold-capture clip remain operator-verified (no Wine/PulseAudio in CI). If the clip persists, it is winepulse playback cold-start (not parec/monitor cooling) and a continuous rolling-buffer `parec` is the deferred next step.
- **`captureReady` variance eliminated by construction (amendment 2026-06-21, persistent capture source).** The per-request `parec` and its readiness gate (Decision 3) and the keep-warm reader (Decision 8) are SUPERSEDED by one long-lived `CaptureSource` (`src/capture.ts`) that the server windows per request — so `captureReady`'s 608↔1765ms swing is gone (replaced by a consistent ~tens-of-ms `windowFirstByteMs`), the always-live capture keeps the start-race solved without a gate, and `/tts` is now SERIALIZED (one window at a time — already required by the shared null sink). The first-Speak clip was diagnosed by the metric, not guessed: `wavMs(2983) > timelineMs(2212)` proved the audio was captured and `trimLeadingSilence` was over-trimming the front (`trimmedMs=284`), so the trim defaults were loosened (threshold 512→150, minRun 4→3, leadInMs 40→80) and the trim is now operator-tunable via `VIVIFY_TRIM_THRESHOLD` / `VIVIFY_TRIM_LEADIN_MS`. Verification boundary: the `CaptureSource` windowing, the `/tts` serialization, the empty-window 500, and the new trim defaults are CI-tested via an INJECTED continuous-capture fake (no Wine/PulseAudio); the real `parec` stability and the opening-word audibility remain operator-verified on rebuild. No `@vivify/core`/browser change; no MS/L&H IP.
- **No third-party IP.** No Microsoft/L&H binaries, `.acs` files, or Wine prefix are involved.
- **New assumed dependency (amendment 2026-06-21):** the architecture now ASSUMES the null-sink monitor streams when idle (true with no `module-suspend-on-idle` loaded). If on some host it does not, the readiness gate fails loudly (a 500 with a clear message) rather than clipping the audio — the honest-failure boundary, operator-verified on rebuild (no Wine/PulseAudio in CI). The gate's cost is surfaced as `captureReady=` in the `[tts-timing]` log.
